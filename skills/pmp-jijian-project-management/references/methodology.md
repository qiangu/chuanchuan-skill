# 方法论（提炼自《极简项目管理》）

目标：用**结构化**降低项目复杂度，把“目标落地、把事办成、让成功可复制”。

## 1) 两个坐标系：五大过程组 × 如来十掌

### 五大过程组（做管理工作的阶段）
- 启动：确立项目合法性与总体要求（目标/授权）
- 规划：把目标具体化，形成路线图与基准（范围/进度/成本）
- 执行：按计划把纸面成果变成现实成果
- 监控：对比基准→找偏差→分析成因→纠偏/变更
- 收尾：验收、关闭、复盘、归档与复用

### 如来十掌（10个相互关联的核心问题/知识领域）
1. 范围：确定工作内容与边界（交付什么/不交付什么）
2. 进度：这些工作何时完成（里程碑、关键路径）
3. 成本：花多大代价完成（投入、预算、成本控制）
4. 质量：做到什么程度才可接受（验收标准）
5. 资源：需要谁/哪些资源来完成（人/设备/工具）
6. 采购：资源不足时如何外包/采购
7. 沟通：内外部如何同步信息与协同
8. 相关方：谁影响成败，如何参与与期望控制
9. 风险：不确定性如何促进/妨碍成功
10. 整合：在相互竞争目标下做最优权衡与决策闭环

建议做法：任何决策/争论卡住时，回到“十掌”逐一补信息；很多问题本质是“某一掌没想清”。

## 2) 启动：先对齐，再开跑

### 2.1 商业论证（是否值得做）
创业/中小企业做项目选择时，至少回答 3 个问题：
- 能不能赚钱（机会/市场/财务）
- 能不能搞得定（技术/产能/约束）
- 对大家有没有好处（风险/环境/社会效益/内部激励）

### 2.2 相关方：权力-利益方格
做法（落地成登记册）：
1) 收集人员资料（姓名/单位/职务/影响范围）
2) 评估期望与态度（支持/中立/反对）
3) 打分：权力、利益（例如 1~10）
4) 放入权力-利益方格
5) 按象限制定策略并持续更新

象限策略（可直接写进沟通计划）：
- 一象限：持续管理，尽量变成支持者
- 二象限：谨慎管理，避免成为反对者（重视其地位与影响）
- 三象限：少投入时间，监控即可
- 四象限：多倾听，往往能提供有用信息

### 2.3 三落实 + RAM/RACI（压实责任）
三落实（降低“旁观者效应”）：
- 任务落实：每项任务**只能有一个负责人**
- 人员落实：责任落实到**具体个人**，不是部门/小组
- 组织落实：人/流程/平台/工具/激励要一致

RAM/RACI 规则要点：
- RACI 表中每项活动**只能有一个 R**
- 责任要经过讨论与一致同意；团队外人员负责的工作，项目经理承担接口与协调责任

### 2.4 一页项目任务书（章程/任务书）
关键不是“写出一页纸”，而是通过起草过程达成**早期共识**，并形成授权。

任务书核心内容（精简版）：
- 可测量目标与成功标准
- 总体要求、概括性描述
- 主要风险
- 总体里程碑进度计划、总体预算
- 审批要求、项目经理职责与权限、发起人/批准人信息

## 3) 规划：WBS → 进度 → 投入 → 风险（形成基准）

### 3.1 WBS（没有 WBS 就没有项目管理）
关键价值：
- 把隐性工作显性化、显性工作结构化、结构工作标准化
- 界定范围、揭示细节，支撑进度/预算/沟通
- 形成可复用的标准 WBS 与 WBS 词典（组织过程资产）

基本规则：
- WBS **面向可交付成果**分解，不是面向活动
- 最底层是工作包；工作包要能定义验收与责任

常见分解方式（按项目选择合适的）：
- 按组成（部件）
- 按功能用途
- 按生命周期阶段
- 按地域/组织（跨边界时常用）

### 3.2 进度：先估算，再排关键路径
估算思路（从粗到细）：
- 类比估算：早期粗估，成本低但误差大
- 参数估算：基于 WBS 分解 + 历史数据 + 单位工作量
- 三点估算：乐观/最可能/悲观（用于风险表达与沟通）

关键路径（CPM）要点：
- 找关键活动（浮动时间≈0），项目周期主要由关键路径决定
- 压缩关键路径常用三招：并行（快速跟进）/增加资源/加班（只对关键路径有效）

### 3.3 成本：看“花多少钱对应完成多少工作量”
预算的本质不是“预计花费的钱”，而是“**计划应完成工作量的价值**”。

最小可执行的核对方式（无需术语也能落地）：
- 对比【计划工作量价值】 vs 【实际完成工作量价值】→ 看进度
- 对比【实际成本】 vs 【实际完成工作量价值】→ 看成本效率
- 只对比【实际花费】 vs 【预算】只能得到“花钱速度”，对项目成败帮助很有限

### 3.4 风险：概率 × 影响（先排序再行动）
概念澄清：
- 已发生：问题（Issue）
- 未发生但可能发生：风险（Risk）
- 风险三要素：事件、概率、影响

做法：
1) 建立影响评级规则 + 概率影响矩阵
2) 识别风险 → 评估概率/影响 → 排序 → 定策略/预案/触发器 → 定 Owner

## 4) 监控：PDCA + 状态报告 + 变更受控

### 4.1 PDCA + “万能公式”
PDCA：Plan / Do / Check / Action（循环迭代）

“万能公式”口诀：
- 行动之前要计划
- 计划之前要分析
- 分析之前要信息

### 4.2 状态报告（最小但必须有）
建议至少包含：
- 本期完成、下期计划
- 当前风险/问题状态
- 偏差分析与预测（时间/成本）
- 本期批准的变更汇总
- 需要决策/升级的事项

### 4.3 变更管理“九步法”（九阴真经）
1) 确认变更事实（先定义问题，避免“用正确方法解决错误问题”）
2) 形成书面变更申请（客户不写就你写，再请其确认）
3) 分析影响：范围/进度/成本/质量/风险等
4) 沟通影响，确认是否取消变更
5) 提出解决方案（至少 3 个：上/中/下策；让管理层做选择题）
6) 查审批权限，选对批准人
7) 召开变更控制会议（必要时），批准/否决
8) 沟通结果并更新基准/计划
9) 执行与跟踪，确保登记-评估-批准-实施闭环

## 5) 收尾：验收、收尾报告、经验教训与归档复用

### 5.1 收尾报告（至少五部分）
1) 原计划概况：范围/进度/成本/质量
2) 偏差说明：范围/进度/成本/质量与计划的差异
3) 客户关系：生命周期中的关键互动与经验
4) 风险总结：主要威胁/机会、应对与结果
5) 经验教训：经过+措施+受益者（要可落实/可度量）

### 5.2 经验教训写法（避免“正确的废话”）
原则：必须具体、可落实、可度量；最好能转成“检查项/标准WBS/标准流程”。

可选机制（组织层面）：用正向激励促进复用（例如内部专利/查阅计费/贡献分成等）。
